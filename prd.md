# Prompt Analyzer & Optimizer — PRD

**Версія:** 0.5  
**Попередня:** 0.4 (Nx + Nest + Angular, vbomba-ui, серверний AI proxy, логування, SQLite, статистика)

---

## 1. Огляд

**Prompt Analyzer & Optimizer** — внутрішній інструмент (з можливістю приватного онлайн-доступу), який допомагає швидко створювати якісні промпти, ітеративно аналізувати їх і отримувати рекомендації для покращення.

**Цінність:**

- Прискорити написання промптів.
- Підвищити їх якість через структурований зворотний зв’язок.
- Навчати користувача через пояснення (explainability).

---

## 2. Цілі та метрики успіху

### 2.1 Основні цілі

- Структурований аналіз промптів (оцінка + розбір за критеріями).
- Автоматичне покращення тексту промпта з поясненням змін.
- Підтримка ітеративного циклу «аналіз → покращення → повтор».

### 2.2 Метрики успіху (MVP)

- Час від введення промпта до першого корисного результату.
- Зміна оцінки після дії «Improve» (до/після).
- Кількість ітерацій на сесію / користувача.
- Внутрішня оцінка задоволеності (опитування / якісний фідбек).

## 3. Цільова аудиторія

| Сегмент | Опис |
|--------|------|
| Розробники | Junior → Senior |
| Нетехнічні користувачі | Потребують простого режиму |
| Power users | Розширений аналіз, деталі |

**Рівні складності контенту:** початковий / середній / просунутий (впливає на глибину пояснень у Advanced mode).

## 4. Функціональні вимоги

### 4.1 Аналіз промпта (FR-1)

- Загальна оцінка в діапазоні **1–10**.
- Розбір за критеріями (окремі нормалізовані бали):
  - Clarity  
  - Context  
  - Specificity  
  - Constraints  
  - Output Format  
- **Час написання (draft duration):** клієнт вимірює тривалість редагування промпта до запуску аналізу і передає значення на backend. Воно входить у контекст для моделі (калібрування: дуже короткий час може вказувати на чернетку тощо) і **впливає на узгоджену загальну оцінку та тон рекомендацій** узгоджено з системними інструкціями — без окремого «ручного» множника в UI, якщо не вирішено інакше в імплементації.

**Прийнятність:** усі критерії присутні в UI; значення узгоджені з відповіддю API (structured output).

### 4.2 Рекомендації (FR-2)

- Список дій: що **додати**, що **прибрати**, що **уточнити**.

### 4.3 Покращення промпта (FR-3)

- Згенерований переписаний промпт.
- Покращена структура та оптимізація під LLM.
- **Explain changes:** що змінено і чому (обов’язково в Advanced; у Simple — скорочено або приховано згідно з режимом).

### 4.4 Режими (FR-4)

| Режим | Поведінка |
|-------|-----------|
| **Simple** | Мінімум деталей, швидкий результат |
| **Advanced** | Повний аналіз, пояснення змін, деталізація |

### 4.5 Історія та версії (FR-5)

- Збереження промптів і результатів аналізу/покращення.
- Ланцюжок версій: original → improved → наступні ітерації (v1, v2, v3…).
- Відстеження ітерацій.
- **Diff між версіями** (візуальне порівняння тексту).
- **Персистентність (MVP):** **сервер** — **SQLite** (історія, версії, метадані користувача після автентифікації). **Клієнт** може використовувати локальне сховище (наприклад IndexedDB) для офлайн-чернеток і швидкого UX; **після логіну** — **автоматична синхронізація** з сервером (злиття / джерело істини на сервері; деталі конфліктів — на етапі імплементації, за замовчуванням пріоритет серверних записів для вже синхронізованих сутностей).

### 4.6 Шаблони (FR-6)

- Базовий набір категорій: **coding**, **marketing**, **fitness**, **general tasks**.
- Шаблон підставляє стартовий текст або структуру в редактор.

### 4.7 Тестування промпта (FR-7, поза MVP)

- Кнопка на кшталт «Show AI response» / симуляція відповіді моделі для швидкої перевірки.
- Не входить у перший реліз MVP; закладається в архітектуру за можливості.

### 4.8 Статистика користувача (FR-8)

- Збір і збереження на **сервері** (SQLite) агрегованих та подієвих метрик: наприклад кількість аналізів / покращень, середні оцінки, час написання до аналізу, ітерації на сесію, помилки API (без змісту промптів).
- Мета: підтримка продуктових метрик (розділ 2), дебаг якості pipeline, майбутні звіти / дашборди (повний UI дашборду — поза обов’язковим MVP, якщо не зафіксовано окремо).

## 5. Користувацький сценарій (основний цикл)

1. Користувач вводить або обирає шаблон і редагує промпт.
2. Запускається аналіз через **backend (Nest)**; на сервер передаються в т. ч. метадані **часу написання** (FR-1).
3. Відображаються: загальна оцінка, breakdown, рекомендації.
4. Користувач натискає «Improve».
5. Показується новий промпт і пояснення змін (залежно від режиму).
6. Збереження в історію; можливість повторити цикл або порівняти версії.

---

## 6. Обробка AI (multi-step pipeline)

Усі виклики до OpenAI виконуються **лише з NestJS** (клієнт не має секретів). Провайдер на етапі продукту: **OpenAI**; абстракція під інші провайдери — за потреби пізніше.

**Продуктова послідовність кроків** (що бачить користувач / контракт API фронту):

1. **Analysis** — оцінка та breakdown.  
2. **Suggestions** — рекомендації.  
3. **Improvement** — переписаний промпт.  
4. **Explain** (переважно Advanced) — пояснення змін.

**Імплементація на сервері (узгоджено для v0.5):** зберегти **чотири змістові кроки** для UX, кешу й спостережуваності; **допускається один LLM-виклик** для кроків **1+2**, якщо відповідь — єдиний structured JSON з блоками analysis і suggestions **без погіршення якості** порівняно з роздільними викликами. Кроки **3** і **4** залишаються **окремими** викликами (Explain може бути скорочено або пропущено в Simple згідно з FR-4). Паралельні виклики між залежними кроками не використовуються.

**Принцип:** structured outputs, ролі на кшталт «prompt engineering expert», мінімізація галюцинацій через жорсткий формат.

## 7. Технологічний стек (цільовий)

| Шар | Технології |
|-----|------------|
| Monorepo | **Nx** — один репозиторій; мінімум застосунків: **Angular (`apps/web`)** + **NestJS (`apps/api`)** |
| Frontend | Angular, `HttpClient`, стан: RxJS / Signals |
| UI / стилізація | Пакет **`vbomba-ui`** з публічного npm registry (залежність **`latest`**) — **єдине джерело** токенів теми та компонентів; попереднє рішення про копію теми з `ecolabel-apps` **скасовано** |
| Backend | **NestJS** — REST (або узгоджений транспорт) до AI proxy, персистентність SQLite, статистика, логування |
| AI | **OpenAI** API, structured outputs (модель — за конфігом середовища, наприклад GPT-4o-mini) |
| Дані | **SQLite** (MVP серверна персистентність) |

**Спільні бібліотеки (Nx libs):** мінімум кілька спільних **TypeScript-інтерфейсів / контрактів** між frontend і backend (DTO запитів/відповідей, коди подій телеметрії); розширення libs (утиліти, валідація) — за потреби без over-engineering на старті.

**Збірка та деплой:** локально — **`nx serve`** для **`apps/web`** і **`apps/api`** (окремі процеси/таргети); для середовищ винесення — **окремі контейнери** (образ для Angular-статики або SSR за архітектурою релізу + образ для Nest).

### 7.1 Логування та кореляція

| Аспект | Вимога |
|--------|--------|
| Backend (Nest) | Структуровані логи **JSON у stdout**; рівні **`info` / `warn` / `error`** (у prod без `debug`, якщо не увімкнено явно) |
| Клієнт | **Структуровані події** (успіх/помилка дії, навігація, латентність викликів до API **без** змісту промптів) — надсилання на backend **ендпоінт телеметрії** або узгоджений механізм |
| Заборона | **Явна заборона** логувати та зберігати в логах **тіло промпта**, повні відповіді моделі та інший високочутливий контент |
| Кореляція | **`requestId`** (або `correlationId`) на ланцюжок **клієнт → Nest → (зовнішні виклики)**; прокидання в заголовках / полях подій для дебагу |

## 8. Автентифікація, доступ і секрети

**Секрети OpenAI**

- **Лише на сервері** (змінні середовища / секрети контейнера Nest). Клієнт **не** містить API-ключів.

**Мережа та еволюція доступу**

- **Поточний етап:** переважно **внутрішня мережа**; CORS і дозволені origin’и налаштовуються під внутрішні хости.
- **План:** **публічний URL** з повноцінною **автентифікацією** перед відкритим доступом.

**Користувачі та auth (дорожня карта)**

- **Спочатку:** один **внутрішній** сценарій доступу (без публічної self-signup або з мінімальним тимчасовим обхідним шляхом — узгодити при імплементації).
- **Далі:** **власні облікові записи** (email + password) та **Google Workspace** (OAuth / SSO за стандартом Nest).

**Rate limiting (Nest)**

- Раціональний дефолт для AI-ендпоїнтів: **до 60 запитів на хвилину** на **користувача** (після появи облікового запису) або на **IP** (до впровадження стійкої ідентифікації), з можливістю зміни через конфіг; окремі жорсткіші ліміти для «важких» маршрутів за потреби.

## 9. Локалізація та мова відповідей AI

**Мови інтерфейсу:** English, Ukrainian, Polish.

**Мова відповідей моделі (аналіз, рекомендації, improve, explain):**

- За замовчуванням — **та сама мова, що обрана для UI**.
- Окремий елемент керування: **«Мова відповіді AI»** (може збігатися з UI або відрізнятися — наприклад UI українською, пояснення англійською).

Запити до API повинні явно враховувати обрану «мову відповіді AI» (system / user instructions або параметр, узгоджений з імплементацією).

## 11. Межі релізу MVP

### 11.1 Обов’язково в MVP (зафіксовано)

- FR-1 Аналіз (оцінка + критерії), з урахуванням **часу написання** (розділ 4.1).  
- FR-2 Рекомендації.  
- FR-3 Покращення + explain (хоча б у Advanced; у Simple — згідно з FR-4).  
- FR-4 Simple / Advanced.  
- FR-5 **Повна історія:** збереження, версії, перегляд ланцюжка, **diff між версіями**; **SQLite на сервері**; локальний кеш/чернетки за потреби; **автосинхронізація після логіну**.  
- FR-6 **Шаблони** (coding, marketing, fitness, general tasks).  
- FR-8 **Статистика користувача** на сервері (мінімум події та агрегати без змісту промптів).  
- Локалізація UI + **перемикач «мова відповіді AI»** (розділ 9).  
- **Nx** monorepo: Angular + NestJS; UI — **vbomba-ui** (`latest`).  
- Виклики OpenAI **тільки через Nest**; ключі **лише на сервері** (розділ 8).  
- **Логування та телеметрія** згідно з §7.1 (у т. ч. `requestId`).  
- **Rate limiting** на Nest згідно з §8.

### 11.2 Поза MVP (наступні ітерації)

- FR-7 Тестування промпта (симуляція відповіді).  
- Повноцінний продуктовий дашборд / розширена аналітика поверх уже зібраної статистики (якщо не введено в MVP окремим рішенням).  
- Learning mode (підказки, навчальні блоки).  
- Інтеграції (IDE, розширення браузера).  
- Публічний URL + **email/password** та **Google Workspace** (коли виходить за межі внутрішнього доступу) — деталізація auth у roadmap, не скасовує обов’язок секретів на сервері.

**UX**
